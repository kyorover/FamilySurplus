# 要件定義書

## 1. システム名称
節約帖(仮)

## 2. プロジェクト概要

### 2.1. 背景と目的
- 一般的な総合型家計簿アプリ（マネーフォワードMEなど）は、全資産や全口座の正確な記録と分析を目的としているため、機能やUIが複雑になりがちである。そのため、日々の入力負荷が高く、継続的な運用に挫折するユーザーが少なくない。
- 本システムは、過去の支出の「緻密な記録」ではなく、日本国内で最も多くを占める一般的な家庭に向けた「小遣い管理」と「家計のゆとり（余剰資金）の可視化」を主目的とする。
- 固定費の管理や複雑な口座連携をあえて排除し、日々のコントロールが可能な「変動費」と「夫婦の小遣い」にターゲットを絞る。
- これにより、家計簿の入力に抵抗がある層でも継続可能な最小限の管理ツールを提供し、世間の生活費平均や物価指数と照らし合わせながら、資産運用などに回せる余剰資金を明確にすることを目的とする。

### 2.2. ターゲットユーザー
- 夫婦の小遣いや日々の変動費を、直感的にストレスなく管理したい一般家庭（日本国内における最大ボリューム層）。
- 世間一般の平均的な生活費や最新の物価指数を知り、自分たちの家計が適正かどうかを客観的に確認したい世帯。
- 細かい家計管理は苦手だが、将来の資産運用（新NISAなど）に回すための「余剰資金」が毎月どれくらいあるのかを把握したい層。
- （実証済みペルソナ：一般的な家計簿アプリでは入力が続かないが、目的が明確に絞られていれば運用可能な主婦層）

### 2.3. コアバリュー・優位性
- **日常に寄り添う資金管理** ：無理な節約を強いるのではなく、小遣いの残額や家計の浮いた分を「ゆとり（資産運用などの余剰資金）」としてポジティブに視覚化する設計。
- **圧倒的なUIのシンプルさ** ：連携遅延やノイズを排除し、手入力における摩擦を極限までゼロに近づけた直感的な操作性。
- **客観的指標による生活費の適正化** ：世間の平均値や物価指数（CPI）という外部の客観的基準を取り入れ、各家庭の世帯構成に合わせた「適正な生活費の目安」を提示する機能。

## 3. 機能要件

### 3.1. 対象とする支出カテゴリと予算設定
- 管理対象を「食費」「外食」「日用品」「養育費」の変動費に限定する。
- 家族（夫婦など）の「小遣い」も主要な設定項目として管理に含める。
- 上記カテゴリに対する「目標金額」をユーザーが任意に設定できる機能を有する。

### 3.2. 日々の入力機能と残額表示
- ユーザーが日々の変動費を手動で入力する機能を提供する。
- 複雑な仕訳を不要とし、カテゴリと金額のみを最短の手数で入力できるUIとする。
- 日々の入力データと設定した目標金額を即座に計算し、リアルタイムに「どれだけ予算（および小遣い）が残っているか」を表示する。

### 3.3. 目安算出・比較機能（外部データ連携）
- ユーザーの「世帯構成（単身、夫婦、子持ち等）」を登録・反映する機能を有する。
- 総務省統計局（e-Stat等）の家計調査データや、最新の消費者物価指数（CPI）をシステム側で取得する。
- 世帯構成、世間の平均値、および物価指数を掛け合わせ、「適正な生活費の目安」を動的に算出し、画面上に表示する。

### 3.4. ゆとり（余剰資金）の可視化機能
- ユーザーが設定した「目標金額」および「適正な生活費の目安」に対する、現在の「予算の残額」と「小遣いの残額」を総合して計算する。
- 計算結果から「家計がどれだけ浮いているか」を算出し、これを資産運用などに回せる「余剰資金」としてダッシュボードに視覚的に大きく表示する。

### 3.5. AIグラフィック・ステータス評価機能
- 予算達成率や外部指標との比較結果から、「家計の状況がどれだけ優れているか」を視覚的に評価・表現する。
- 評価結果（ステータス）に応じたアイコンやグラフィックを生成AIを用いて動的に出力し、画面上に表示する。
- 生成時のAPIコストと遅延（レイテンシ）を抑えるため、ステータスの組み合わせパターンに応じた画像を事前生成してストックするか、初回生成時にキャッシュして使い回す仕組みを必須とする。

## 4. 非機能要件・アーキテクチャ設計

### 4.1. インフラストラクチャ方針
- マイクロ法人での運用および初期の無課金（無料枠）運用を前提とし、アクセス増に応じた自動拡張（オートスケール）を実現するため、AWSを利用した「フルサーバーレス・アーキテクチャ」を採用する。

### 4.2. システム構成要素
- **フロントエンド（アプリ）** ：Flutter または React Native を採用し、iOS/Android向けクロスプラットフォーム開発を行う。
- **認証基盤** ：Amazon Cognito を利用し、ユーザー登録・ログイン機能を提供する。
- **APIゲートウェイ** ：Amazon API Gateway を利用し、アプリからのリクエストを処理する。
- **バックエンド（処理）** ：AWS Lambda を利用し、余剰金計算や比較ロジックを実行する。
- **データベース** ：Amazon DynamoDB（NoSQL）を採用し、柔軟なデータ構造と高速な読み書きを実現する。
- **外部データ取得バッチ** ：Amazon EventBridge と Lambda を組み合わせ、e-Stat等の統計データを定期的に自動取得・保存する。
- **AI画像生成・配信** ：Amazon Bedrock または OpenAI API 等を利用して画像を生成し、Amazon S3 に保存の上、CloudFront（CDN）経由で高速配信する。

## 5. AI駆動開発（バイブコーディング）特化型構造設計ルール
AI（Gemini等）のハルシネーションを防止し、自動出力の確実性を高め、トークン消費を最小化するために、以下の構造的ルールを厳格に適用して開発を進める。
複雑なドメインモデリング（DDD等）やテスト駆動開発（TDD）などの重厚な設計手法は避け、ラピッドプロトタイピングとフレームワーク先行での構築を徹底する。

### 5.1. 完全なフラット設計（複雑な階層構造の排除）
- ファイルを複数層に分割するアーキテクチャは採用しない。
- 「1つのAPIエンドポイント」や「1つの機能」に対して、コントローラーの受け口からデータベースへの読み書きまでを、単一の短いファイル内で完結（垂直分割）させる。

### 5.2. ロジックの「純粋関数」としての隔離
- 「目標金額と日々の支出からの余剰資金の算出」や、「世間平均と物価指数による目安の補正」などの計算処理は、データベース通信やUI描画を含まない独立した純粋関数として別ファイル（ディレクトリ例：`/functions`）に切り出す。
- AIにはこの計算処理のみを生成させることで、バグの混入を防ぐ。

### 5.3. UIのマイクロ・コンポーネント化
- フロントエンドの画面は1ファイルにまとめず、極小のパーツ単位で構造化する。
- 「予算残額表示部品（RemainingBudgetDisplay）」「金額入力パッド（ExpenseInputPad）」など、単一の責務を持つウィジェットごとにファイルを分割し、AIへの指示を局所化する。

### 5.4. 型定義（スキーマ）の中央集権化
- データベースの項目名やデータ構造の不整合を防ぐため、システム全体で利用する型定義（User, Budget, Expense等のスキーマ）は1つの専用ファイルに集約する。
- AIにコードを生成させる際は、必ずこの型定義ファイルをプロンプトの前提条件として読み込ませる運用とする。
