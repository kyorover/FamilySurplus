# 節約帖 サブスクリプション課金導入 要件・方式設計書

## 1. 全体方針と提供価値
初期構築コストと運用・保守コストを最小化し、税務・審査・セキュリティ上のリスクを排除するため、以下の基本方針とする。

- **決済基盤**: Apple App Store (IAP) および Google Play Billing を利用。
- **状態管理**: RevenueCat によるフルマネージドのレシート検証と状態管理。
- **プラン設計**: 自動更新サブスクリプションの「月額プラン」「年額プラン」のみ。買い切り（一括払い）はインフラコスト増大リスクを考慮し非採用とする。
- **認証連携**: 既存の AWS Cognito のユーザーID（`sub`）をRevenueCatと紐付け、セキュアに管理する。

## 2. リスク対策とコンプライアンス要件

### A. 会計・税務および検証時の課金リスク回避
- **テスト方針**: 開発者および関係者の実クレジットカードによる本番決済は行わない。
- **環境**: App Store Connect の「Sandboxテスター」および Google Play Console の「ライセンステスト」機能のみを用いて検証を行う。これにより、実費の発生およびテストデータによる架空売上（税務リスク）を完全に排除する。
- **売上処理**: 本番リリース後の売上および消費税の代理徴収は、すべてプラットフォーム（Apple/Google）のレポートを正として会計処理（freee等）を行う。

### B. ストア審査（リジェクト）対策要件
アプリ内課金画面（Paywall）には、Apple/Googleのガイドラインに抵触しないよう以下のUI要素を必須とする。
1. プラン（月額/年額）ごとの明確な価格と更新期間の表示
2. 「自動更新される旨」および「期間終了の24時間前までに解約が必要な旨」の明記
3. OS標準の解約画面（サブスクリプション管理）への導線テキスト
4. 「購入の復元（Restore）」ボタンの設置
5. 特商法に基づく表記、利用規約、プライバシーポリシーへのリンク

## 3. システムアーキテクチャ要件

### A. フロントエンド (Expo / React Native)
- **ライブラリ**: `react-native-purchases` (RevenueCat SDK) を導入。
- **開発環境**: Expo Go はネイティブSDK非対応のため、EAS Build を用いて `expo-dev-client` による開発用カスタムビルドを作成し、実機検証を行う。

### B. バックエンド (AWS API Gateway + Lambda + DynamoDB)
- **真の権限管理**: フロントエンドの改ざんを防ぐため、プレミアム権限の正本はバックエンド（DynamoDB）に持たせる。
- **状態同期**: RevenueCatからの Webhook（`INITIAL_PURCHASE`, `RENEWAL`, `CANCELLATION`, `EXPIRATION` 等）を受け取る Lambda エンドポイントを構築し、非同期で DynamoDB のユーザー設定を更新する。

## 4. 影響範囲（スキーマとファイル）

### A. 型定義 (`src/types.ts`)
`UserSettings` またはそれに準ずるユーザー情報型に以下を追加する提案（実装時に確定）。
- `subscriptionStatus`: `'active' | 'expired' | 'none'`
- `premiumExpiresAt`: `string | null`

### B. DynamoDB (Settings テーブル等)
- `revenueCatAppUserId` (String): Cognito `sub` と一致
- `subscriptionStatus` (String): 現在のステータス
- `entitlementId` (String): 付与権限（例: `premium`）

### C. 新規・改修コンポーネント
- `src/components/subscription/SubscriptionPaywallModal.tsx`: プラン選択・購入・復元UI
- `src/screens/SettingsScreen.tsx`: サブスク管理・特商法リンクの追加
- `hesokuri-backend/lambda/controllers/webhookController.ts`: Webhook受信用Lambda
- `hesokuri-backend/lib/hesokuri-backend-stack.ts`: Webhook用API Gatewayの定義追加

## 5. 実装・導入ステップ（ロードマップ）

### フェーズ 1: インフラ・コンソール事前準備（コード記述なし）
1. App Store Connect および Google Play Console にてアプリを登録。
2. 両コンソールで「自動更新サブスクリプション（月額・年額）」のプロダクトを作成。
3. Sandboxテスターアカウント、およびライセンステスト用アカウントの作成。
4. RevenueCat ダッシュボードにてプロジェクトを作成し、上記ストア設定と連携（APIキー取得、Entitlementの作成）。

### フェーズ 2: 開発環境構築とフロントエンド実装
1. Expoプロジェクトに `expo-dev-client`, `react-native-purchases` をインストール。
2. `app.config.ts` を修正し、IAP（In-App Purchase）のパーミッションを追加。
3. EAS Build (`eas build --profile development`) を実行し、実機用Dev Clientを作成。
4. アプリ起動時にCognitoの `sub` を用いてRevenueCatを初期化（`LogIn`）するロジックの実装。
5. Paywall（課金画面）UIおよび設定画面の実装。

### フェーズ 3: バックエンド Webhook の実装
1. AWS CDK (`hesokuri-backend`) に Webhook受信用エンドポイントを定義。
2. Lambda関数を実装し、RevenueCatからの署名検証およびDynamoDBのステータス更新処理を構築。
3. RevenueCat ダッシュボードに AWS API Gateway のURLをWebhook宛先として登録。

### フェーズ 4: 法務および審査提出
1. リポジトリ内ドキュメント（`利用規約.md`, `プライバシーポリシー.md`）の課金対応アップデート。
2. マイクロ法人の情報を記載した「特定商取引法に基づく表記」ページのWeb公開とアプリ内リンク設置。
3. Sandbox環境での完全な状態遷移（購入、更新、期限切れ、リストア）テスト完了後、本番審査へ提出。
