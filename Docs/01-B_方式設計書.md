# 方式設計書

## 1. システムアーキテクチャ概要
本システムは、日本国内の一般的な家庭に向けた「小遣い管理」および「家計のゆとり（余剰資金）の可視化」を目的とするモバイルアプリケーションである。
マイクロ法人での運用を前提とし、初期の運用コストを極限まで抑える（無課金スタート）こと、および将来的なアクセス増加に伴う自動スケーリングを実現するため、AWSを利用した「フルサーバーレス・アーキテクチャ」を採用する。
また、AI（Gemini等）によるコード自動生成（バイブコーディング）を最大限に活用するため、複雑な階層構造を持たないフラットな設計とし、フロントエンドからバックエンドまで一貫して型定義（TypeScript）を共有する構成とする。

---

## 2. 技術スタック選定

### 2.1. フロントエンド
- **フレームワーク** : React Native + Expo
- **言語** : TypeScript
- **選定理由** : 既存のWeb/JS資産を活用でき、迅速なMVP（Minimum Viable Product）開発に極めて有利であるため。また、厳格な型定義（TypeScript）を用いることで、AIにスキーマを与えやすく、UIコンポーネントの自動生成におけるハルシネーションを構造的に防ぐことができる。

### 2.2. バックエンド・API層
- **APIゲートウェイ** : Amazon API Gateway (HTTP API)
- **コンピューティング** : AWS Lambda
- **言語/ランタイム** : Node.js / TypeScript
- **選定理由** : サーバーを常時稼働させないため、リクエスト時のみの課金となり無料枠に収めやすい。API GatewayのHTTP APIは軽量かつ低コストであり、Lambdaとの統合が容易である。TypeScriptで記述することで、フロントエンドと型定義を完全に共有する。

### 2.3. データストア
- **データベース** : Amazon DynamoDB (NoSQL)
- **選定理由** : オンデマンドキャパシティモードを利用することで、アクセスが少ない初期段階のコストをほぼゼロに抑えることができる。アクセスパターンを事前に定義したシンプルなキー設計（パーティションキーとソートキー）により、月次集計などのパフォーマンスを高速に維持する。

### 2.4. インフラストラクチャ・CI/CD
- **IaC (Infrastructure as Code)** : Terraform または AWS CDK
- **CI/CD** : GitHub Actions
- **選定理由** : 初期段階からAWSの最小構成をコード化しておくことで、インフラとアプリケーションの連携設定をAIに正確に把握させることができる。

## 3. インフラストラクチャ設計（フルサーバーレス構成）

### 3.1. トラフィックフローおよびシステム連携
- **ユーザー認証** : アプリケーション起動時、Amazon Cognitoを通じてユーザーの認証・認可を行う。発行されたJWT（JSON Web Token）を用いて後続のAPIリクエストを行う。
- **データ入力・取得** : アプリケーションからのリクエストは、Amazon API Gateway (HTTP API) で受け付ける。ここでJWTの検証を行い、正当なリクエストのみをAWS Lambdaへルーティングする。
- **ビジネスロジックの実行** : AWS Lambdaがリクエストを処理し、計算ロジック（純粋関数）を実行した上で、Amazon DynamoDBに対してデータの読み書きを行う。
- **外部データ連携（非同期）** : Amazon EventBridgeのスケジュール機能を用いて定期的にLambdaを起動し、総務省統計局（e-Stat）のAPI等から世間平均や消費者物価指数（CPI）を取得、DynamoDBに保存する。

### 3.2. コスト最適化戦略（無課金スタート運用）
- 各AWSサービスの無料枠（Lambdaの月間100万リクエスト、API Gatewayの無料枠、DynamoDBの25GBストレージなど）を最大限に活用する。
- AWS Budgetsを設定し、無料枠の85%に到達した時点で管理者にアラートを通知する「ゼロコスト監視」を導入する。

## 4. アプリケーション構造設計（バイブコーディング特化型）

### 4.1. 設計思想（ラピッドプロトタイピングの徹底）
- AIによるコード出力の確実性を高め、トークンの浪費を防ぐため、システム全体の階層は完全にフラットな設計とする。
- 複雑なドメイン駆動設計（DDD）や、開発速度を低下させる過度なテスト駆動開発（TDD）といった重厚な手法は意図的に排除する。まずはインフラとアプリケーションのフレームワーク（骨組み）の構築を先行させ、そこにUIと必要な機能を実装していくラピッドプロトタイピング手法を厳格に採用する。

### 4.2. 中央集権型スキーマ管理（型定義）
- システム全体で利用するデータ構造（User, Budget, Expense, IndicatorData 等）は、単一の型定義ファイル（例: `types.ts`）に集約する。
- AIにコードを生成させる際は、常にこの型定義ファイルをプロンプトの前提条件として読み込ませることで、フロントエンドとバックエンド間でのデータ不整合を完全に防止する。

### 4.3. ロジックの純粋関数化
- 「日々の支出と目標金額から余剰資金を算出する処理」や「物価指数を用いて適正な生活費目安を補正する処理」は、データベース通信やUI描画といった副作用を持たない「純粋関数」として独立したディレクトリ（例: `/functions`）に切り出す。
- 自動テスト（ユニットテスト）は、この独立した計算ロジックの入出力チェックのみに限定し、システム全体に対する過剰なテスト実装は行わない。

### 4.4. UIのマイクロ・コンポーネント化
- フロントエンドの画面は、極小のパーツ単位（例: `RemainingBudgetDisplay`, `ExpenseInputPad`）に細分化してコンポーネント化する。
- AIへの指示は「この小さなコンポーネントの見た目と挙動のみ」に局所化し、1ファイルあたりのコード行数を抑えることでハルシネーションを防ぐ。

## 5. AI画像生成および配信・キャッシュ戦略

### 5.1. 画像生成フロー
- 家計のステータス（予算達成率、余剰資金の有無など）に応じたグラフィック生成には、OpenAI API、Amazon Bedrock、またはAnthropic等の生成AI APIを利用する。
- セキュリティとAPIキーの保護のため、画像生成のAPI呼び出しは必ずバックエンド（AWS Lambda）側で実行する。

### 5.2. キャッシュおよびストレージ戦略
生成AIのAPI呼び出しに伴う従量課金コストの増大と、数秒に及ぶレイテンシ（遅延）を回避するため、以下のキャッシュ戦略を必須とする。
- **事前生成（パターンストック）** : ステータスの組み合わせパターンがある程度限定できる場合は、バッチ処理等であらかじめ画像を生成し、Amazon S3に保存しておく。
- **初回生成キャッシュ** : 動的なプロンプトにより初回のみ画像を生成し、その結果をAmazon S3に保存する。
- **CDN配信** : アプリケーションからの画像読み込みは、すべてAmazon CloudFront（CDN）を経由させる。S3に保存された画像（キャッシュ）をエッジロケーションから高速に配信し、同一ステータスに対するAI APIの重複呼び出しを完全に遮断する。

## 6. 外部データ連携設計

### 6.1. 政府統計（e-Stat）および物価指数（CPI）の取得バッチ
- 世間一般の平均的な生活費や最新の物価指数をシステムに取り込むため、総務省統計局が提供する「e-Stat API」を利用する。
- データの更新頻度（月次等）に合わせ、Amazon EventBridgeで定期的にトリガーされる専用のAWS Lambda関数を実装する。
- 取得したデータはシステム内部で扱いやすい形式（JSON等）に整形し、DynamoDBの専用テーブル（または項目）へ保存・更新する。アプリケーション側は、外部APIへ直接アクセスするのではなく、常にDynamoDB内の最新データを参照する。

## 7. セキュリティおよび運用監視設計

### 7.1. 認証・認可
- Amazon Cognitoのユーザープールを利用し、セキュアなサインアップ・サインイン処理を実装する。
- バックエンドAPIへのアクセスは、API GatewayにおけるJWTオーソライザーを用いて厳密にアクセス制御を行う。AIが生成するコードには、ビジネスロジックの実装のみを委譲し、認証・認可の仕組みはフレームワーク側で担保する。

### 7.2. 機密情報管理
- 外部AIサービス（OpenAI等）のAPIキーや、e-StatのアプリケーションIDなどの機密情報は、ソースコード上にハードコードしない。
- AWS Secrets Manager または AWS Systems Manager Parameter Store（SecureString）を利用して暗号化・管理し、Lambda実行時に環境変数として安全に読み込む。

### 7.3. コストおよびパフォーマンス監視
- Amazon CloudWatchを利用し、Lambdaの実行回数、エラー率、およびAPI Gatewayのレイテンシを監視する。
- Amazon S3のデータ転送料金や生成AI APIの呼び出し回数をモニタリングし、設定した閾値を超過した際のアラートを徹底することで、予期せぬインフラコストの急増を防止する。
