## 🏃‍♂️ 今後の実装順位と残タスク

既存の純粋関数やデータベースへの影響（デグレリスク）が少ないUI/フロントエンド層の改修から順次着手し、その後バックエンド・インフラ層の構築へと進むセオリーに基づき優先順位を設定しています。

| 優先順位 | 大項目 | タスク名 | 具体的な仕様・制約（推測排除） | 影響するファイル / 追加要素 |
| :---: | :--- | :--- | :--- | :--- |
| **1** | **課金・広告** | 2-1. スキーマ拡張 | `AccountInfo` に課金プランと有効期限を追加。**※収入(income等)の概念・入力は絶対に追加しない。** | 型: `src/types.ts` (`subscriptionPlan` の拡張など) |
| **2** | **課金・広告** | 2-2. DBカラム追加 (DynamoDB) | 既存のアカウントテーブルに対し、上記属性を追加保存する。 | AWS: `AccountsTable` (PKに対して属性追加のみ。GSI不要) |
| **3** | **課金・広告** | 2-3. ダッシュボード広告実装 | ユーザーのアカウント情報（無課金）を判定し、ダッシュボード上にバナー広告（AdMob等）を表示。 | 設定: `app.json` (SDK設定)<br>UI: `DashboardScreen.tsx` (広告コンポーネント配置) |
| **4** | **インフラ** | 3-1. 環境分離（Dev/Prod）の構成 | CDKを用いて開発（Dev）と本番（Prod）のスタックを分離。API GatewayやDynamoDBを物理的に分ける。 | AWS: `bin/hesokuri-backend.ts`<br>AWS: `lib/hesokuri-backend-stack.ts` |
| **5** | **インフラ** | 3-2. 自動デプロイ（CI/CD）の構築 | GitHub Actions等を導入し、mainマージをトリガーとして自動で `npx cdk deploy` が実行されるパイプライン構築。 | 新規: `.github/workflows/deploy.yml` |
