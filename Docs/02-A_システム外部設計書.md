# 節約帖（FamilySurplus）システム外部設計書

## 1. システム全体概要
「節約帖」は、各家庭の支出実績と設定予算を比較・評価し、浮いた予算を「へそくり（余剰金）」として可視化・管理するアプリケーションである。
本システムにおける最大の制約・特徴として**「システム上で収入（給与・ボーナス等）の入力・保持・評価を一切行わない」**ことが挙げられる。あくまで「支出」と「予算」の差分管理に特化し、ユーザーの心理的ハードルを下げつつ、ゲーム感覚（ガーデン機能）で節約を促進する。

## 2. プロジェクト（構成要素）の概要と責務

システムは大きく分けて以下の3つの論理コンポーネント（プロジェクト群）で構成される。

### 2.1. Frontend (`hesokuri-cho`)
* **技術スタック**: React Native, Expo, TypeScript, Zustand
* **概要と存在目的**: 
  ユーザーが直接操作するiOS/Android向けモバイルアプリケーション。
  ユーザーの支出入力、予算設定、ダッシュボードでのへそくり状況確認、ガーデン（ご褒美UI）の描画を担当する。
* **主要な責務**:
  * ZustandによるUI状態管理とAPI通信のキャッシュ。
  * ユーザー入力のバリデーション。
  * 取得した「自家のデータ」と「統計データ（キャッシュ済）」を用いた、純粋関数による予算評価スコアリング（外部通信を持たない）。

### 2.2. Backend API (`hesokuri-backend` API Gateway群)
* **技術スタック**: AWS CDK, API Gateway, Lambda, DynamoDB, Amazon Cognito
* **概要と存在目的**: 
  フロントエンドからの同期的なリクエスト（データ取得・更新）を処理する自社管理のコアAPI群。
  セキュアな認証基盤とデータ永続化層を提供する。
* **主要な責務**:
  * Cognitoによるユーザー認証・トークン発行。
  * 支出記録、家族設定、ガーデン配置データ等のDynamoDBに対するCRUD処理。
  * **キャッシュ済み統計データの提供**: アプリ起動時に要求される最新の物価指数や平均生活費を、外部APIを叩かずに即座（DynamoDBから）返却する。

### 2.3. Batch Lambda (`hesokuri-backend` EventBridge起動群)
* **技術スタック**: AWS EventBridge (Scheduler), Lambda, Node.js (fetch API)
* **概要と存在目的**: 
  ユーザーの操作とは非同期に、定期的にバックグラウンドで実行されるシステム保守・データ収集用のプログラム群。
* **主要な責務**:
  * 外部機関（e-Stat等）からの統計データの定期取得。
  * アプリ側が利用しやすいフォーマットへのデータ整形と、DynamoDBへのキャッシュ保存。

## 3. 外部統計データ取得 Lambda バッチ設計

予算評価の精度を向上させるため、外部の公的統計データ（物価指数、世帯人数別の平均支出）を毎月自動取得・保存するバッチ処理を定義する。

### 3.1. バッチ基本情報
* **関数名**: `FetchNationalStatisticsBatch`
* **トリガー**: AWS EventBridge 
* **実行頻度**: 毎月1日の午前3:00 (JST) `cron(0 18 L * ? *)` ※UTC指定
* **タイムアウト**: 30秒

### 3.2. 処理フロー
1. **シークレット取得**: 
   AWS Secrets Manager から e-Stat 等の外部APIアクセス用アプリケーションID（APIキー）を取得する。
2. **外部通信 (e-Stat API)**:
   * **データA**: 消費者物価指数（CPI）の最新月データ。
   * **データB**: 家計調査（家計収支編）における「世帯人数別の消費支出（食費、光熱費など固定費中心）」データ。※収入データは除外。
3. **データ整形**:
   取得した複雑なXML/JSONをパースし、アプリ側で利用する `NationalStatistics` 型のJSONオブジェクトに変換する。
4. **DynamoDBへ保存（キャッシュ更新）**:
   * **テーブル**: `SystemConfigTable`（または既存テーブルの別パーティション）
   * **PK**: `CONFIG#STATISTICS`
   * **SK**: `MONTH#YYYY-MM` (取得対象月)
   * **内容**: 整形済みの統計データオブジェクト。

### 3.3. エラーハンドリングとリトライ
* 外部APIのメンテナンス等で通信に失敗した場合、Lambda側の組み込みリトライ（非同期呼び出しの場合は最大2回）を実施。
* 最終的に失敗した場合は AWS SNS を経由して管理者（開発チーム）にアラートメールを送信し、前月のキャッシュデータを継続利用するフェールセーフ設計とする。
* ※ この処理が失敗しても、フロントエンド（アプリ）の起動やユーザーの通常操作には一切影響を与えない。
